* DONE [#A] Repair allocator and runtime memory safety
Effort: H
Goal: Remove the native memory-management defects that can corrupt process state or write past configured buffers, while preserving successful runtime behavior and current interfaces.
Notes: Start here. Several later cleanups depend on the allocator and buffered IO being trustworthy. Keep public APIs and success-path outputs unchanged. Build first with `make linux-lib linux-exe CCACHE=1`. Prefer adding narrow regressions over broad rewrites.

** DONE [#A] Fix sfpool free-list corruption and platform allocation checks
Why: The internal pool currently zeroes a block after writing the next-pointer into it, which can destroy the free list. It also treats `mmap()` failures like `NULL`, which is wrong on POSIX and can hide allocator setup failures.
Change: In `src/sfpool.h`, reorder secure wiping and free-list relinking so zeroing never erases allocator metadata. Audit `sfpool_malloc`, `sfpool_free`, and `sfpool_realloc` together so pool blocks, fallback allocations, and profiling counters remain internally consistent. Replace `NULL` checks for `mmap()` with `MAP_FAILED`, and verify the Apple and POSIX secure-lock bookkeeping reflects the actual lock outcome. Keep the existing pool layout and fallback policy unless a local change is required to preserve correctness.
Tests: Rebuild with `make linux-lib linux-exe CCACHE=1`. Run `bats test/lua/crypto.bats` because it exercises octet-heavy allocations. Add a focused regression under `test/lua` or `test/api` that repeatedly allocates and frees octets large enough to hit both pool and fallback paths, then confirm it fails before the fix and passes after it.
Done when: Pool allocations can be freed and reused repeatedly without corrupting future allocations, `mmap()` setup errors are detected correctly, and the touched regression suite passes.

** DONE [#A] Make zenroom init and teardown own every resource on every error path
Why: `zen_init()` currently has partial construction branches that leak or skip teardown when config parsing, RNG setup, Lua creation, or pool setup fail. That makes the main runtime harder to maintain and unsafe to refactor.
Change: In `src/zenroom.c`, make allocation and cleanup order explicit. Check every allocation result, including the initial `zenroom_t`, RNG allocation, and pool allocation. Route every failure through one local cleanup path that releases only the resources already acquired. Preserve all current defaults, globals, and success-path side effects. Do not change script execution behavior or log format selection.
Tests: Rebuild with `make linux-lib linux-exe CCACHE=1`. Run `bats test/zencode/bindings_exec.bats` and `bats test/zencode/bindings_logfmt.bats`. Add one focused regression in `test/api` or `test/shell` that forces an init failure through bad config and confirms the process exits cleanly without crashing or double-freeing.
Done when: Every init failure path returns cleanly without leaks or partial teardown hazards, and the binding executor tests still pass unchanged.

** DONE [#A] Stop buffered stdout and stderr writes from overrunning user buffers
Why: The buffered print and log paths detect “no space left” but continue into `memcpy` and terminator writes anyway. That defeats the purpose of bounded host buffers and makes debugging memory damage harder.
Change: In `src/zen_io.c`, convert the output-buffer checks in `zen_print`, `zen_write`, `printerr`, and `zen_log` into hard early exits on overflow. Preserve the current error reporting text if practical. Audit nearby length arithmetic so newline, tab, JSON quote/comma, and final `'\0'` bytes are all included in the bound checks exactly once.
Tests: Rebuild with `make linux-lib linux-exe CCACHE=1`. Run `bats test/zencode/bindings_logfmt.bats` and `bats test/zencode/bindings_exec.bats`. Add one focused regression that installs a deliberately small host output buffer and confirms Zenroom reports the buffer exhaustion without writing past it.
Done when: No buffered output path writes after signaling overflow, length accounting is explicit, and binding-level output tests still pass.

* WIP [#A] Harden external input boundaries and fixed-size buffers

** DONE [#A] Fix zencode-exec base64 decoding and configuration append bounds
Why: The binding executor allocates one byte too few for decoded lines and appends `,logfmt=json` into a fixed config buffer without proving there is space. This is an input-driven overflow risk in a boundary executable.
Change: In `src/zencode-exec.c`, make `line_alloc()` reserve space for a terminator and validate decode lengths before writing them. Replace the `strcat()` config append with explicit bounded assembly. Also make the line-1 config read detect truncation correctly instead of relying on `strlen(conf) >= MAX_CONFIG`, which `fgets()` cannot produce.
Tests: Rebuild with `make linux-lib linux-exe CCACHE=1`. Run `bats test/zencode/bindings_exec.bats` and `bats test/zencode/bindings_logfmt.bats`. Add regressions for: maximum-length valid config input, invalid non-base64 input, and a decoded line whose exact decoded size reaches the current boundary.
Done when: Every input line is decoded into a correctly sized buffer, config assembly is bounded, and the existing binding executor tests remain green.

** DONE [#A] Correct config parser validation and length handling
Why: The config parser’s `rngseed` validation skips part of the payload, and the `maxmem` path reuses `STR_MAXITER_LEN` implicitly. This is small code, but it defines trusted runtime limits and determinism settings.
Change: In `src/zen_config.c`, rewrite the `rngseed`, `maxiter`, and `maxmem` validation branches so lengths, prefixes, and digit loops are obvious and correct. Introduce a dedicated `STR_MAXMEM_LEN` constant in `src/zenroom.h` if the code benefits from one, but keep the user-visible configuration syntax unchanged. While editing, remove dead commented frees and make the finite-state transitions easier to follow without changing accepted valid configurations.
Tests: Rebuild with `make linux-lib linux-exe CCACHE=1`. Run `bats test/zencode/bindings_logfmt.bats`. Add focused negative tests for malformed `rngseed`, malformed `maxiter`, and malformed `maxmem` inputs, plus one valid boundary-length test for each.
Done when: Config validation checks the full payload it copies, length constants match their target fields, and the binding-logfmt suite plus new regressions pass.

** TODO [#A] Replace unsafe DER and PEM string handling in x509 helpers
Why: Several X.509 helpers treat DER slices as C strings and rely on `%s`/`sprintf()` over binary data. That is unsafe, misreads embedded NULs, and obscures ASN.1 parsing logic.
Change: In `src/zen_x509.c`, replace string-format extraction with explicit length-based copies and `lua_pushlstring()` where appropriate. Preserve the current returned table shape and field names. Harden SAN parsing so short or malformed extension contents fail safely instead of reading through unchecked lengths. Keep `pem_to_base64()` behavior unchanged for valid PEM input, but make its extraction flow easier to read.
Tests: Rebuild with `make linux-lib linux-exe CCACHE=1`. Run `bats test/lua/crypto.bats` because it includes `x509.lua`. Add focused regressions for SAN parsing with embedded NUL-adjacent data, malformed extension lengths, and a standard PEM import/export roundtrip.
Done when: No X.509 helper formats binary blobs as C strings, malformed certificate slices fail safely, and the existing X.509 Lua tests still pass.

** TODO [#A] Harden Longfellow native string and attribute boundaries
Why: The Longfellow bridge currently truncates table-derived values silently, mixes octet buffers with APIs that expect C strings, and allocates fixed public-key string buffers without making the contract explicit. That is both a safety risk and a maintenance risk at a boundary to external code.
Change: In `src/zen_longfellow.c`, make every string boundary explicit. Replace silent truncation in `_get_kv()` and `_get_attributes()` with either validated bounded copies or explicit failures that match current caller expectations. Introduce local staging buffers for values passed as `const char *` to the external prover and verifier APIs so those calls always receive NUL-terminated strings with checked lengths. Keep the current Lua table shape, argument order, and successful proof/verify behavior unchanged.
Tests: Rebuild with `make linux-lib linux-exe CCACHE=1`. Run `bats test/lua/crypto.bats`, which includes `longfellow_guards.lua`. Add focused regressions for oversized attribute ids or values, malformed `now` and `doc_type` string inputs, and a valid prove/verify roundtrip using the current fixture path.
Done when: Longfellow boundary crossings use validated lengths and explicit string staging, oversized values fail predictably instead of truncating silently, and the existing Longfellow tests still pass.

Tests: Rebuild with `make linux-lib linux-exe CCACHE=1`. Run `bats test/lua/crypto.bats` and, if needed, add small stress regressions under `test/lua/native` or `test/lua` that repeatedly create, copy, and garbage-collect these userdata types.
